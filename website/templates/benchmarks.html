When to Use What

ScenarioRecommendedWhy
Real-time / streaming Cleora Fastest for re-embedding. Constant memory. Incremental updates.
Million-node graphs Cleora Embeds 2M nodes in 4s with 1.9 GB. Best scale factor (19x from 4k to 2M).
Quick baseline RandNE Sub-second on small graphs. Scales to millions but with higher memory cost.
Best accuracy (medium graphs) Cleora / NetMF Cleora leads on Facebook ego network. NetMF close behind with higher memory cost.
Homophily tuning Node2Vec p/q parameters control BFS vs DFS exploration. Best for graphs under 50k nodes.
Memory constrained Cleora 4 MB for 4k-node graph. 1.9 GB for 2M nodes. Best memory efficiency at every scale.
Maximum embedding speed Cleora Use propagation="symmetric" for the fastest variant — trades ~5% accuracy for a small speed gain on regular graphs.

Methodology

  • Task: Node classification with 80/20 train-test split (where labels available)
  • Labels: Community labels via Louvain community detection (SNAP edge-list datasets have no native labels)
  • Classifier: Nearest Centroid (no hyperparameters)
  • Embedding dimension: 1024 for all algorithms
  • Metrics: Accuracy, Macro F1, wall-clock time, peak memory delta
  • Hardware: Single CPU core, no GPU
  • Datasets: Benchmarks use synthetically generated graphs (Erdős–Rényi, Barabási–Albert, SBM) with node and edge counts matching named real-world datasets. PPI-large, Flickr, ogbn-arxiv, and Yelp are simulated at matching scale; ego-Facebook, roadNet-CA, and soc-LiveJournal1 are downloaded from SNAP. Synthetic graphs are not the original datasets — they reproduce scale and community structure, not content.
  • Walk-based params: num_walks=10, walk_length=20 (Facebook); excluded for larger graphs
  • Excluded algorithms: GraRep and HOPE (require dense n×n matrices, infeasible for 4k+ nodes)
Reproduce: Run python3 benchmark_report.py from the pycleora repository to regenerate all results. LiveJournal requires >16 GB RAM and may take significant time for graph construction.
